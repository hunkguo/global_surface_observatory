# 机场天气采集项目开发说明书

## 1. 项目目标

本项目旨在构建一个可扩展的机场天气采集系统，核心目标如下：

1. 获取全球机场站点代码与基础元数据
2. 基于机场代码获取航空气象报文，包括：
   - METAR（实时观测）
   - TAF（机场预报）
3. 同时接入多个数据源，保留原始报文与标准化结果
4. 将站点信息与天气数据写入 SQLite 数据库，供后续分析、回测与市场监控使用

该项目适用于：
- 航空气象数据研究
- 天气预测市场监控
- 机场实时天气看板
- 多源观测数据交叉验证
- 自动化定价/预警系统的数据底座

---

## 2. 功能范围

### 2.1 第一阶段：获取全球机场代码

目标：建立全球机场站点主表。

需要获取的信息包括：
- ICAO 代码（优先）
- IATA 代码（若有）
- 机场名称
- 城市
- 国家/地区
- 纬度
- 经度
- 海拔
- 时区（可选）
- 是否可获取 METAR/TAF（后续验证补充）

说明：
- 航空气象系统中，**ICAO 代码**是最核心的站点标识
- 后续的 METAR / TAF 抓取，原则上统一以 ICAO 为主键
- IATA 更多用于展示或和外部机场列表映射

---

### 2.2 第二阶段：按机场代码抓取天气

目标：针对机场代码批量获取航空气象数据。

要采集的数据类型：
- METAR
- TAF
- 后续可扩展：SPECI / SIGMET / AIRMET / PIREP

基础能力要求：
- 单机场抓取
- 批量机场抓取
- 定时刷新
- 原始报文留存
- 关键字段标准化提取
- 多数据源比对与回退

---

### 2.3 第三阶段：多数据源接入

目标：提高覆盖率、稳定性和数据验证能力。

建议接入的数据源：

#### 数据源 A：Aviation Weather Center (AWC)
- 角色：官方主数据源之一
- 适用：METAR / TAF
- 优点：权威、稳定、适合程序化访问
- 缺点：不同地区覆盖细节可能不完全一致

#### 数据源 B：AVWX
- 角色：开发友好的聚合型航空气象 API
- 适用：METAR / TAF
- 优点：返回结构化 JSON，解码方便
- 缺点：通常需要 API Key，受第三方服务可用性影响

#### 数据源 C：其他补充源（预留）
可选：
- NOAA / NWS
- Ogimet
- Iowa State Mesonet
- 各国气象局开放接口
- 机场或空管公开站点

这些补充源主要用于：
- 数据交叉验证
- 特定国家/地区补漏
- 结算规则验证

---

### 2.4 第四阶段：写入 SQLite

目标：构建本地轻量数据库，支持检索、对比、回测和策略开发。

数据库至少要保存：
- 机场主数据
- 数据源信息
- 每次抓取任务记录
- 原始报文
- 标准化天气结果

---

## 4. 开发分解

## 4.1 模块一：全球机场代码获取

### 目标
建立全球机场基础元数据表，为后续天气抓取提供站点列表。

### 输入
可选来源：
- 开放机场数据库（如 OurAirports）
- 开放航空数据 CSV/JSON
- 第三方机场 API

### 输出
输出文件：
- `shared_data/airports.json`

输出字段示例：

```json
{
  "icao": "RJTT",
  "iata": "HND",
  "name": "Tokyo Haneda Airport",
  "city": "Tokyo",
  "country": "JP",
  "latitude": 35.553,
  "longitude": 139.781,
  "elevation_ft": 21
}
```

### 关键要求
- 去重
- 标准化 ICAO 大写
- 过滤无 ICAO 的记录
- 保留原始来源标识

### 建议脚本
- `scripts/fetch_airport_codes.py`

### 后续增强
- 增加国家过滤
- 增加是否支持 METAR/TAF 探测字段
- 增加城市到机场站点候选映射

---

## 4.2 模块二：基于机场代码抓取 METAR + TAF

### 目标
对指定 ICAO 站点获取实时观测与预报。

### 输入
- 单个 ICAO 代码，例如：`RJTT`
- 多个 ICAO 代码列表
- 或全量机场站点清单

### 输出
输出文件：
- `shared_data/aviation_weather_latest.json`
- `shared_data/aviation_weather_history/YYYY-MM-DD/*.json`

### 统一输出结构示例

```json
{
  "city": "Tokyo",
  "country": "JP",
  "station_code": "RJTT",
  "metar": {
    "source": "awc",
    "raw": "METAR RJTT 061230Z 20013KT CAVOK 18/12 Q1010 NOSIG",
    "time": "2026-04-06T12:30:00Z",
    "temperature_c": 18,
    "dewpoint_c": 12,
    "wind_speed": 13,
    "visibility": "6+"
  },
  "taf": {
    "source": "awc",
    "raw": "TAF RJTT ...",
    "time": "2026-04-06T11:05:00Z"
  }
}
```

### 标准化字段建议
对于 METAR，建议抽取：
- station_code
- report_time
- temperature_c
- dewpoint_c
- wind_dir
- wind_speed
- gust
- visibility
- altimeter
- flight_category
- raw_text

对于 TAF，建议抽取：
- station_code
- issue_time
- valid_from
- valid_to
- raw_text
- 结构化 forecast 列表

---

## 4.3 模块三：多数据源采集策略

### 设计目标
对于每个机场代码，系统应尝试从多个源获取同一类数据。

### 推荐抓取顺序

#### 方案一：主源 + 回退
1. 优先抓 AWC
2. 如果失败，再抓 AVWX
3. 如果仍失败，记录错误并写入任务日志

#### 方案二：双源同时抓取
1. 同时抓 AWC
2. 同时抓 AVWX
3. 保留两份原始结果
4. 比较字段差异

建议：
- **研究阶段：双源同时抓**
- **生产阶段：主源 + 备用回退**

### 为什么要保留多源
- 可以验证站点覆盖是否一致
- 可以检查报文同步延迟
- 可以比较解析结果差异
- 可以避免单源故障导致系统停摆

## 6. 数据流设计

整体流程如下：

1. 获取全球机场代码
2. 清洗后写入 `airports` 表
3. 生成待抓取机场列表
4. 对每个机场从多个源获取 METAR + TAF
5. 标准化字段
6. 写入 `metar_reports` / `taf_reports`
7. 记录抓取任务结果到 `fetch_runs`

流程图（简化）：

```text
开放机场数据
    ↓
fetch_airport_codes.py
    ↓
airports.json
    ↓
SQLite.airports
    ↓
fetch_aviation_weather.py
    ↓
AWC / AVWX / 其他补充源
    ↓
标准化解析
    ↓
aviation_weather_latest.json
    ↓
sync_aviation_to_sqlite.py
    ↓
SQLite (metar_reports / taf_reports / fetch_runs)
```

## 9. 非功能要求

### 稳定性
- 单个机场失败不影响全局抓取
- 单个源失败可回退到其他源
- 数据写库失败要记录日志

### 可维护性
- 数据源客户端独立封装
- 解析逻辑与存储逻辑解耦
- 表结构支持后续扩展 SPECI / SIGMET

### 可追溯性
- 所有原始报文保留
- 所有原始 JSON 尽量保留
- 每次抓取有 run_id

### 性能
- 支持批量抓取
- 控制 API 频率
- 后续可加入异步并发

## 10. 风险与注意事项

### 10.1 ICAO 与城市映射误差
同一城市可能有多个机场，不能直接把城市温度等同于某一个机场温度。

### 10.2 结算规则与数据源不一致
如果后续用于天气预测市场，必须确认市场结算用的是哪个官方站点，而不是仅凭机场直觉选站。

### 10.3 不同源更新时间不同
同一时刻 AWC 与 AVWX 返回的数据可能存在数分钟差异，属于正常现象。

### 10.4 原始报文解析不一致
不同源可能对 METAR/TAF 做不同程度解码，因此必须保留 raw text。

## 12. 当前建议结论

如果以“快速落地”为原则，推荐如下最小开发路径：

1. 用开放机场数据库建立 ICAO 主表
2. 用 AWC + AVWX 双源抓 METAR/TAF
3. 原始报文 + 标准化字段同时保存
4. 全部写入 SQLite
5. 后续再增加 SPECI、历史回放和市场结算映射

也就是说，本项目的第一版，不需要一开始就追求复杂模型，而是先把这条数据链打通：

**机场代码 → 多源气象报文 → 标准化 → SQLite 落库**

这条链一旦稳定，后面无论是做看板、回测、告警，还是做天气市场监控，都会轻松很多。
